# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository structure

Two independent packages, each with their own `node_modules` and `package.json`:

- `app/` — React + Vite frontend (the application under test)
- `functional-tests/` — Playwright E2E test suite

## Commands

### App (`app/`)
```bash
npm run dev      # start dev server at http://localhost:5173
npm run build    # production build
npm run lint     # oxlint
```

### Functional tests (`functional-tests/`)
```bash
npm test                                           # run all tests (auto-starts app dev server)
npx playwright test tests/tasks/toggling-tasks.spec.ts  # run a single spec file
npx playwright test --debug                        # run with Playwright inspector
```

Tests run against Chromium only. The config always starts `npm run dev` in `../app`; locally it reuses an existing server on port 5173 if one is already running, in CI it always starts a fresh one.

## App architecture

Single-page React app with no backend — all state is in-memory and resets on page load.

- `App.jsx` — top-level router: renders `LoginPage` or `TasksPage` based on `user` state
- `LoginPage.jsx` — hardcoded user store (`alice@example.com / password123`, `bob@example.com / hunter2`); sets user on success
- `TasksPage.jsx` — all task logic lives here: `INITIAL_TASKS` (IDs 1–3) seeded on every load, `nextId` counter starts at 4 for new tasks

There is no component library, router, or state management library.

## Functional test architecture

Tests use the Page Object Model pattern with Playwright fixtures.

**Layer overview:**
```
tests/          ← spec files (import from @fixtures/fixtures)
fixtures/       ← Playwright fixture definitions
  login.ts      ← navigates to / and exposes LoginPage
  task-manager.ts ← depends on loginPage; logs in as alice, waits for task list, exposes TaskManagerPage
  fixtures.ts   ← composes fixtures into a single extended test object
model/pages/    ← Page Object classes
  base.ts       ← BasePage with protected page: Page
  login.ts      ← LoginPage (locators + goto/login methods)
  task-manager.ts ← TaskManagerPage (locators, stat counts, filter buttons, task item methods)
```

**Path aliases** (defined in `tsconfig.json`):
- `@fixtures/*` → `functional-tests/fixtures/*`
- `@model/*` → `functional-tests/model/*`

**Key design decisions:**
- `taskManagerPage` fixture handles login automatically — tests that use it start already authenticated as alice
- `loginPage` fixture is available for tests that need to exercise the login flow itself
- Task items are located by title (`taskManagerPage.taskCheckbox('Set up Playwright')`) rather than by internal ID, so tests work for both seeded and newly added tasks
- Stat count locators (`activeCount`, `completedCount`, `totalCount`) point to the `.stat-number` child element, so assertions use `toHaveText('2')` directly

**Test organisation:** spec files live under `tests/tasks/`; non-feature smoke tests at `tests/` root.

**Test plan:** `functional-tests/specs/task-management.plan.md` — the full scenario list generated by the `playwright-test-planner` agent. Not all scenarios are intended to be implemented (see memory for the agreed subset).
